# Faithful Document Translation Skill

中译英翻译，保留原文格式，严格遵循术语表。

## 核心原则

1. **术语表优先** — 所有在术语表中出现的中文词，必须严格按照术语表的英文翻译，不得自行翻译
2. **格式分层保留** — 三类格式规则有优先级之分：
   - **视觉样式（保留）**：加粗、斜体、下划线、标黄底色（yellow highlight）、划线、画框、对齐方式 — 必须完全复现到译文
   - **背景底纹（清除）**：段落或文字的背景灰底/纹理填充（w:shd，如 F7F7F7 等）— 不保留，统一为白底黑字。灰底在扫描后会斑驳不清，正式提交文件要求白纸黑字
   - **字体与字号（覆盖）**：不管原文用什么字体字号，译文正文一律按下方规范强制执行 Times New Roman 12pt
3. **正式文件语气** — 英文译文使用正式、专业的书面文件语气

## 工作流程

### 第一步：加载术语表

用户提供术语表文件（.xlsx 或 .csv），格式为：
- 第一列：中文术语
- 第二列：英文翻译
- 中间可能有分类合并行（忽略分类行，只取术语和翻译）

读取后建立术语映射字典。

**如果用户没有提供术语表**，直接询问"请提供术语表"。

### 第二步：分析源文档

支持 Word 文档（.docx）和 PDF 文档（.pdf）：
- 读取所有段落、表格、样式信息
- 记录每个文本块的样式：标黄、划线、加粗、画框、斜体等
- 表格内容单独提取，保持单元格结构
- 检查引号风格（弯引号还是直引号）

**如果文档是扫描件PDF**，告知用户需提供可编辑版本。

### 第三步：预处理（关键）

#### 3.1 引号归一化

**中文源文档中的引号风格不统一是翻译字典匹配失败的首要原因。**

在开始翻译之前，将源文档和翻译字典的所有 key 统一为同一种引号风格（推荐弯引号 U+201C/U+201D，因为 Word 默认生成弯引号）：

- 弯引号 `""`（U+201C/U+201D）→ 保持不变
- 直引号 `""`（U+0022）→ 转为弯引号
- 中文书名号 `《》` 不受影响

Python 实践：统一使用单引号 `'...'` 包裹字典 key，内部用弯引号，从根本上避免引号冲突。生成翻译字典后，逐条检查源文档中是否存在对应 key，提前发现不匹配。

**关键：引号归一化必须在每一个 dictionary lookup 入口执行。** 包括初始翻译、批量翻译后的二次修正、残留中文修复等所有阶段。如果发现某个单元格翻译失败但内容含引号，第一反应应是引号不匹配——不要假设翻译字典中的 key 和单元格内容用的引号是同一个字符。在 lookup 函数入口处统一做一次 normalize_quotes() 处理。

#### 3.2 OCR 识别误差检测与修正

**源文档如果有OCR识别误差，翻译字典 key 必须以校正后的正确文本为准。**

检测重点：
1. **年份数字丢失**：扫描 `20\d{2}年` 模式，如 `203年`、`205年` 等 3 位数年份
2. **常见形近字**：`没/设`、`枝/技`、`曰/日`、`右/有`、`合/回` 等
3. **标点误识**：长横被拆为连续短横、中点变句点

修正方法：对每个疑似 OCR 误差，用搜索工具检查源目录中其他文件相同位置是否使用正确字，交叉验证后改正。翻译字典 key 必须使用修正后的文本。完整形近字对照表见 `references/ocr-errors.md`。

#### 3.3 方括号中文标签检测

**中文文档中常出现被 Word 自动拆分为多个 run 的方括号标签**（如 `【图片】` 被拆成 `【` + `图片` + `】` 三个 run），导致按 run 级别替换时无法匹配完整标签。

在开始翻译之前，扫描源文档是否存在以下方括号中文标签：

| 中文标签 | 英文替换 |
|---------|---------|
| `【图片】` | `[Image]` |
| `【条形码】` | `[Barcode]` |
| `【图标】` | `[Logo]` |
| `【盖章】` | `[Seal]` |
| `【英文材料无需翻译】` | `[No Translation Required — Source Material in English]` |

翻译完成后，对输出文档调用 `fix_cn_labels(doc)` 进行段落级替换（检测 `para.text.strip()` 完整字符串，不按 run 匹配）。此函数已内置在 `translate_engine.py` 中。

#### 3.4 日期预处理

**中文日期必须整体转换为英文格式，禁止逐字直译。**

使用 `convert_chinese_dates()` 对所有待翻译文本做预处理：
- `2025年10月22日` → `October 22, 2025`
- `2025年10月` → `October 2025`
- `10月22日` → `October 22`

`translate_text()` 内部已自动调用此函数。Claude 翻译正文时也应在翻译前自行检查并处理中文日期，**禁止产出 "2025 October 22 day" 或 "2025Year10Month22Day" 等中式直译**。

#### 3.5 机械错误预防

`clean_mechanical_errors()` 已内置在 `translate_text()` 和 `clear_and_set_text()` 中，自动修复：
- 拼写重复（`decdecision` → `decision`）
- 单词粘连（`ImprovementResearch` → `Improvement Research`）
- 标点缺空格（`climate.trend` → `climate. trend`）
- 重复词（`University University` → `University`）

Claude 翻译正文时同样需要避免这些机械错误。

### 第四步：翻译与格式应用

#### 4.1 术语匹配策略

**只用最长匹配优先（Longest-Match First）**，不用简单的逐条顺序替换。短术语先匹配会破坏长术语——例如"品宅装饰科技"和"内装"同时存在时，"内装"先替换，"品宅装饰科技"就永远无法匹配。

具体做法：将所有术语按长度降序排列，从文本第一个字符开始依次检查匹配，匹配成功则跳转。同时：

- **版本号模糊匹配**：术语表有"XXX系统 V1.0"（V前有空格），原文是"XXX系统V1.0"（无空格），应能归一化后匹配
- **英文文本自保护**：原文中已有的英文（如"CARR"），不要通过中文翻译再去重复替换

实现代码见 `references/translation-engine.md`。

#### 4.2 通用术语库

常见公司名、职位名、奖项名、技术术语等如果不在用户术语表里，使用内置通用术语库自动翻译。不确定的词返回列表询问用户。

#### 4.3 翻译策略

**核心原则：不省略、不自由发挥、逐句忠实翻译。产出完全通顺的英文是基本职责，不存在"保留中文等用户处理"的选项。**

根据内容类型选择策略：

**策略A：词典匹配（用于结构化数据）**
适用于非完整句子的单一术语、固定短语、表格标签：
- 表头：`序号 → No.`、`工程名称 → Project Name`
- 类别标签：`优秀工程勘测一等奖 → Outstanding Engineering Survey — First Prize`
- 页脚/固定文本：`秘书处 → Secretariat`

使用 `translate_engine.py` 的 `translate_text()` 做最长匹配替换。**不能用于完整句子**——中文英文语序完全不同（定语位置、修饰关系、中心词都变了），逐词替换必然产生中英夹杂的碎片。

**策略B：理解式翻译（用于叙述性文本）**
适用于所有完整句子——法律条文、新闻报道、通知公告、管理办法等。约束规则：
1. **逐句对照** — 保持原文的条款编号、段落结构一一对应，不合并不拆分
2. **不省略** — 每个信息点（数字、日期、条件状语、限定词等）都要在译文中体现
3. **不自由发挥** — 不添加原文没有的内容，不改变原文语义范围和语气
4. **术语表锁死** — 术语表已有的词严格使用其英文翻译，不自创译法
5. **不确定的专有名词** — 译文中保留中文原文在括号内，如 `Beijing Huadian Project Co., Ltd.（北京华电工程有限公司）`
6. **输出形态** — 独立完整的英文，不含中文残留

**策略选择规则：**
```
当前处理的内容是什么？
  ├─ 非完整句子（表头、标签、页脚、单个术语）
  │   └─ 策略A：词典匹配
  ├─ 完整句子 + 非专有名词（法律条款、新闻报道、通知等）
  │   └─ 策略B：理解式翻译（严格逐句、不省略）
  ├─ 表格单元格 + 中英混合文本
  │   └─ 策略C：混合单元格翻译（见下方）
  └─ 完整句子 + 专有名词（项目名称、公司名称、成果名称）
      └─ 保持中文不动（无法翻译的专名）
```

**策略C：混合单元格翻译（用于表格中已有部分英文的单元格）**

表格中常出现中英混杂的单元格：部分行已翻译、部分行未翻译，或同一行内英中文混杂（如 `构建新型电力系统of淮北电网"十四五"规划研究`、`China Renewable Energy Engineering Institute,\n华东勘测设计研究院有限公司白鹤滩水力发电厂\nHarbin Electric Machinery Co., Ltd.\n中国三峡建工（集团）有限公司`）。

处理规则：
1. **以完整单元格为单位翻译**，不拆分后逐行 lookup。拆分会导致各行独立匹配不到全量上下文。
2. **保留已有英文**，只翻译中文部分。不能把已有的英文当作未翻译文本重新处理。
3. **嵌入式英文词**（如 `of`、`Hydropower`、`Thermal Power`、`New Energy Projects`）要作为整体保留，不能拆开处理。`构建新型电力系统of淮北电网` → `Building a New Power System for Huaibei Power Grid`（of 自然融入英文语序）。
4. **引号项目名**（如 `"十四五"`、`"上大压小"`、`"盐光互补"`）在翻译时整体处理：`"十四五"规划` → `"14th Five-Year" Plan`，`"上大压小"` → `"Suppressing Small and Replacing with Large"`。弯引号保持弯引号。
5. **多行组织名**：不同行是不同机构时，保持换行分隔，每行独立成完整英文机构名。
6. **不可直接拼接**：不能简单把中文词组替换成英文单词就完事（那会产生`构建新型电力系统 of Huaibei Power Grid "14th Five-Year" 规划研究`这种碎片英文）。必须以完整通顺的英文句子/短语为输出目标。

#### 4.4 格式处理

**格式保留规则：**

| 原文格式 | 译文处理 |
|---------|---------|
| 标黄底色 | 英文文字同样标黄 |
| 划线 | 英文文字同样加划线 |
| 加粗 | 英文文字同样加粗 |
| 加黑/阴影 | ⛔ 清除背景纹理，白纸黑字 |
| 画框 | 英文文字同样加框 |
| 斜体 | 书名号、电影名等用斜体 |
| 右对齐 | 签名/日期区域保持右对齐 |
| 居中 | 标题类内容保持居中 |

**特殊内容处理：**
- 证据编号：`Exhibit #` 始终置于英文标题之前（正式文件 Exhibit 格式要求，不移动编号位置）
- 版本号：放最后，如 `Technical Achievement Name V1.0`
- 盖章/签名：`(Seal)` / `(Signature)`
- 书名号：《书名》→ *Book Title*（斜体）

**字体与间距规范（覆盖原文）：**
- 正文（Normal 样式）：Times New Roman 12pt，**2倍行距**。疏朗行距方便审阅者在纸面手写批注
- 标题：保持原有标题样式，不强制指定字体字号
- 表格内文字：Times New Roman 12pt，**1倍行距**，段前段后为 0。表格空间有限，需紧凑以容纳更多内容

**实现方式：** 使用 `clear_and_set_text()` 方法：先复制源格式（保留标黄/加粗等）→ 清除底纹 → 强制 TNR 12pt。实现细节见 `references/translation-engine.md`。

### 第五步：质量验证 — 残留中文检查

翻译完成后，**必须**对输出的所有 .docx 文件执行残留中文扫描：

1. 遍历每个输出文件的全部段落和表格，使用正则 `[一-鿿　-〿＀-￯]` 检测中文字符
2. 若发现任何中文，打印文件路径、段落号、中文字符数及上下文
3. 逐一排查原因并修复。残留中文常见原因排查顺序：
   - **引号不匹配**（最高概率）：检查单元格内容中的引号字符是否和翻译字典 key 一致。用 normalize_quotes() 统一后再匹配
   - **方括号标签拆分**（第二高概率）：`【图片】`/`【条形码】` 等被 Word 拆为多个 run，按 run 匹配不到。调用 `fix_cn_labels(doc)` 做段落级检测替换
   - **格式共识规则**：若标黄/粗体在译文中错误，检查源文档该段落的 run 拆分情况——标黄丢失意味着所有带标黄的 run 不在第一个 run 中（引擎已修复为共识规则）；粗体过度意味着第一个 run 有粗体但其余 run 没有（引擎已修复为全粗体才保留）
   - **中英混合单元格**：单元格内已有部分英文（如部分行已翻译），整段文本并非纯中文，因此没被提取翻译。需要用策略C以完整单元格为单位处理
   - **段落拆分不一致**：原文分段方式和字典 key 不一致（如有无换行、空格差异）
   - **OCR 误差**：识别错误的字导致 lookup 失败
   - **翻译字典按段落索引映射**：不在字典中的段落原封不动保留中文。期刊页眉、标题残留、图表标题延续行等边缘段落最容易被遗漏。翻译后全文档 CN 扫描是唯一安全网
4. 重新翻译，直到所有段落无中文残留

**不能省略此步骤。** 中文残留可能来自引号不匹配、段落拆分、OCR 误差等多类原因，肉眼难以发现。

### 第六步：翻译后审核校对

残留中文检查通过后，对译文进行**逐段英文质量审核**。目标是让译文读起来像自然、流畅的学术英语，而非机器直译。

**审核检查清单：**

| 类别 | 检查项 | 示例 |
|---|---|---|
| 语法 | 主谓一致、时态一致、冠词缺失 | `the project include three part` → `the project includes three parts` |
| 用词 | 搭配不当、中式选词 | `put forward` → `proposed` / `introduced` |
| 句式 | 杂糅句、逗号拼接、缺少连接词 | 拆分超长句或合并过短句 |
| 翻译腔 | 逐字对应、中文语序直译 | `has very important meaning` → `is of significant importance` |
| 机械错误 | 重复字母、粘连词、标点缺空格 | 见第三步 3.5 节的列表 |
| 日期格式 | 中文日期未转换 | `2025年10月` → 必须为 `October 2025` |

**处理方式：** 对每段译文逐一检查，发现问题直接修改。不要输出"发现 X 个问题请确认"——作为翻译审校，直接给出修正后的最终版本才是完整交付。

**特别注意——策略B的完整英文质量：** 使用策略B翻译的叙述性文本（法律条文、技术描述等）是质量审核的重中之重，因为这些内容是审阅者判断英文水平的依据。必须逐句确认：
1. 没有中文直译痕迹（如 "in the condition of" 替代 "under"）
2. 句子之间有自然的逻辑连接（however, therefore, furthermore）
3. 专业术语统一（同一概念全文档用同一个英文词）

### 第七步：输出文档

- 格式：.docx
- 文件名：`原文名_EN.docx`
- 存储位置：用户指定

## 格式说明

处理过程中向用户说明：
- 术语表加载了多少条术语
- 遇到术语表外的生僻词时，说明译法
- 如果某些格式无法保留，说明原因

## 错误处理

- 术语表格式错误 → 询问用户正确的列位置
- 文档读取失败 → 说明失败原因，提供替代方案
- 术语冲突（同一中文有多个英文）→ 使用最常用或最正式的版本

## 正文翻译协议

所有正文内容均由 Claude 直接翻译，按第四步策略B的规则执行（逐句对照、不省略、不自由发挥、术语表锁死）。

**例外：** 如果用户明确表示"这部分我有现成英文稿"，则直接录入用户提供的英文。
